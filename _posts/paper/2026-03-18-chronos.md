# 0. 论文信息
- 标题：Chronos: Temporal-Aware Conversational Agents with Structured Event Retrieval for Long-Term Memory
- 链接：<https://arxiv.org/abs/2603.16862>（v1, 2026-03-17）
- 论文类型：arXiv preprint（cs.CL）
- 关键词：Long-term conversational memory / temporal reasoning / event extraction / agentic retrieval

# 1. 这篇在解决什么问题？
长时跨度（跨周/月）的对话记忆里，用户问题经常不是“我说过什么”，而是：
- **时间约束**：上个月/最近/那次旅行后一周
- **更新追踪**：偏好/事实被多次修改，需要选“最新的值”
- **跨会话聚合**：某类事件在某段时间出现了多少次
- **多跳**：先找到事件 A 的时间点，再围绕它二次检索

作者认为：
- 仅 turn-level retrieval（检索原始对话 turns）缺少可执行的时间结构；
- 纯 KG/全量事实抽取太重，且容易引入与当前问题无关的“context entropy”。

# 2. 核心思路（一句话）
**双索引（turn calendar + event calendar）**：把对话里“能落到时间轴上的事情”抽成结构化事件用于时间过滤与聚合，同时保留原始 turns 作为语义上下文；查询时再用 **dynamic prompting** 给 agent 生成检索策略，让它通过工具调用循环完成多跳检索。

# 3. 方法（可复现视角）
下面按“要自己复现一个 Chronos”来写：数据结构、索引流程、查询流程、关键超参。

## 3.1 两个日历（两个索引）
### A) Turn Calendar（原始对话索引）
存储单位：conversation exchange/turn（论文里以“turns/exchanges”描述）。

建议最小字段（复现可用）：
```json
{
  "session_id": "...",
  "turn_id": "...",
  "t_conv": "2025-01-02T10:03:00+08:00",
  "speaker": "user|assistant",
  "text": "...原始文本..."
}
```
向量索引用 embedding(text)。

### B) Event Calendar（结构化事件索引）
事件抽取条件：必须有 \<subject, verb, object\> 三元组。

建议最小字段：
```json
{
  "session_id": "...",
  "source_turn_ids": ["..."],
  "event_id": "...",
  "svo": {"subj": "...", "verb": "...", "obj": "..."},
  "start_datetime": "...ISO8601...",
  "end_datetime": "...ISO8601...",
  "aliases": ["...", "..."],
  "raw_evidence": "可选：触发该事件的原句"
}
```
论文要点：
- 时间是 **datetime range**（start/end）而不是单点；
- 对模糊表达（recently/last month 等）用 t_conv 计算合适的时间窗；
- 为每个事件生成 **2–4 个 lexical aliases**（完全不同词汇）增强关键词检索/grep 召回。

> 注：论文未给出严格的时间窗映射规则表（例如“recently”=几天）。复现时需要你自己给一套可解释规则或再去看作者代码（若公开）。

## 3.2 Indexing：事件抽取与入库
### Step 1：分 chunk
- 每个 session：最多 **25 turns** 一批
- 批与批之间 **5-turn overlap**

### Step 2：对每个 chunk 调用 LLM 做 event extraction
输入：一段 turns + 每个 turn 的 t_conv。
输出：事件列表（每个事件带 SVO + datetime range + aliases）。

关键实现点（来自论文描述）：
- 只抽“temporally-grounded state transitions / timestamped occurrences”（时间上有意义的发生/状态改变），避免全量事实图谱。
- 多分辨率时间归一：显式日期原样保留；相对/模糊时间结合 t_conv 解析为区间。

### Step 3：分别入库
- Turn Calendar：embed turn text → vector index
- Event Calendar：embed（建议将 svo + aliases + 时间字段串起来）→ vector index

论文报告的 embedding：**text-embedding-3-large**。

## 3.3 Query-time：dynamic prompting + initial retrieval + agent loop
Chronos 的查询是“三段式”：先给 agent 一个比较强的 turn 种子上下文，再让 agent 工具调用在两条索引里补证据。

### 3.3.1 Dynamic Prompting（每题生成“检索计划”）
输入：用户问题 q。
输出：1–5 条 bullet，告诉 agent：
- 要找哪些实体/属性
- 时间范围/过滤条件
- 是否需要多跳（先找参考事件，再基于时间窗口检索）

论文说明模板生成用一个小模型：**Gemini 3 Flash**。

> 论文没有在正文/附录给出 meta-prompt 的逐字文本，只给了“它要求输出什么”的描述 + 示例输出。

### 3.3.2 Initial Retrieval（先从 Turn Calendar 预取上下文）
论文的固定 pipeline（重要，能复现）：
1. Turn Calendar dense retrieval：top **100**（cosine similarity）
2. Cross-encoder rerank：**Cohere Rerank v3** 对这 100 重排
3. 取 top **15** turns
4. context expansion：对每条 turn 加上同 session 中的**前 1 条 + 后 1 条**
5. 格式化：按 session date 分组输出（“Session 1 (YYYY-MM-DD)”）

### 3.3.3 Chronos Agent（工具调用循环）
agent 以 ReAct 的“思考—调用—观察”循环工作。

提供两类工具、两条索引共四个接口：
- `search_turns(query, top_k)` / `search_events(query, top_k)`：向量检索
- `grep_turns(keyword)` / `grep_events(keyword)`：精确匹配（本地文件/文本）

论文中特别强调：
- grep 用来补 embedding miss（罕见实体/精确短语）
- vector 用来补语义泛化
- agent 可以“迭代性”缩小时间窗、交叉对照 event 与原始对话

# 4. 实验（把能复现的数字写全）
## 4.1 Benchmark：LongMemEvalS
- 总题数：500
- 6 类任务：
  - KU（knowledge updates）
  - MS（multi-session aggregation）
  - SSA（single-session assistant recall）
  - SSP（single-session preference recall）
  - SSU（single-session user recall）
  - TR（temporal reasoning）

## 4.2 实验设置（论文写死的配置）
- Embedding：text-embedding-3-large
- Temperature：0（所有 generation calls）
- Rerank：Cohere Rerank v3
- API：商业 API 标准 tier

两种系统配置：
- **Chronos Low**：generation model = GPT-4o
- **Chronos High**：generation model = Claude Opus 4.6

## 4.3 结果（整体 + 分类型准确率）
**Chronos Low：92.60%**
- KU 96.15
- MS 91.73
- SSA 100.00
- SSP 80.00
- SSU 94.29
- TR 90.23

**Chronos High：95.60%**
- KU 100.00
- MS 88.72
- SSA 100.00
- SSP 100.00
- SSU 98.57
- TR 95.50

# 6. Ablation（复现时最该看的“必要组件”）
从 Table 3 可以直接读出“哪些组件是必须的”：
- 对 Chronos Low：
  - 去掉 initial retrieval：93.1 → 76.7（-16.4）
  - 去掉 dynamic prompting：93.1 → 78.4（-14.7；其中 SSP 掉到 42.9）
  - 去掉 events（Turns Only）：93.1 → 58.6（-34.5，几乎腰斩）
- 对 Chronos High：
  - dynamic prompting 几乎不影响 overall（94.8 → 94.8）
  - 但检索形态限制（grep-only/vector-only）仍显著掉点

# 7. 限制（工程复现时会踩的坑）
论文的限制写得很工程：
1. **存储**：双索引（turn+event）占空间比 turn-only 大。
2. **离线成本**：事件抽取需要 LLM indexing（但通过 25-turn batch + overlap 缓解）。
3. **在线复杂度**：查询时并行检索 turns/events + agent loop 增加延迟；作者用 initial retrieval 减少重复检索。

# 8. Prompt（论文里能拿到多少？）
- Dynamic prompting：论文描述存在“meta-prompt”，但**未提供逐字模板**；只说明它要求抽取 targets（entities/attributes/temporal constraints/operations），输出 1–5 条检索指导，并给了一个示例输出。
- Event extraction：同样没有逐字 prompt，只给出抽取条件（必须 SVO）、输出字段（start/end datetime range）、alias 数量（2–4）与示例（Fitbit）。

> 如果你要完全复刻论文效果，下一步通常需要：作者代码/附录 prompt、或自行设计 prompt 并做 ablation。

# 9. 复现清单（Checklist）
如果你要从 0 复现一个“可跑的 Chronos”，最低需要：
- [ ] 数据：多 session 对话 + 每条 turn 时间戳 t_conv
- [ ] LLM event extractor：输出 SVO + datetime range + aliases
- [ ] 两个向量索引：turn/event
- [ ] initial retrieval pipeline（100→rerank→15→±1 扩展）
- [ ] dynamic prompting generator（每题 1–5 bullets）
- [ ] agent loop：支持对两索引做 vector search + grep
- [ ] 评估：LongMemEvalS judge（或同等判分协议）

# 10. 我的判断（值不值得抄/改什么）
**最值得抄的点**：
- “只结构化时间相关事件，而不是全量 KG”的折中很实用。
- dynamic prompting 把 query rewrite 升级成“检索计划”，在多类型 query（KU/MS/TR/Preference）场景更稳。

**我会改的点**：
- 给 temporal normalization 一个明确的、可审计的规则表（否则难 debug）。
- 对 event extraction 加质量控制：抽取一致性、重复合并、跨 chunk overlap 的去重策略。
- 对 agent loop 做预算控制（最大 tool calls / 最大检索窗口 / 失败回退策略）。

---

一句话结论：Chronos 的核心不是“更强的 RAG”，而是把长期对话记忆里最难的部分（时间）做成可过滤的事件索引，再把检索策略做成每题自适应（dynamic prompting），最后用 agent loop 把多跳问题跑通；在 LongMemEvalS 上给出很硬的准确率支撑。
